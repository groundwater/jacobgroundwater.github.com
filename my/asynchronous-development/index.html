Asynchronous Development

Posted May 21, 2012 by Jacob Groundwater & filed under Journals.

I have been reading about asynchronous development recently, it’s a very interesting angle for software development. You will either love or hate the idea. I haven’t tried it yet, but I will be using it for my next project.

Before reading the bullet points, I want you to apply suspension of disbelief for a moment. This does work. It cannot be applied callously, but there are many successful teams using it.

Selling Points

  • No meetings, ever
    • Okay obviously there is face-to-face communication, but what this really means is no mandatory meetings. Any meeting should be worth the time of every person involved.
    • Meetings tend to be ad-hoc, involve fewer people, and focus on details rather than big picture
  • Work when you want, where you want, how you want
    • Success or failure is judged by working product, not time spent in the office
    • No time accounting, sick days are not recorded

Reality Check

This seems great, but devils advocates are likely to point out how this is liable to all fall apart. Unchecked, I would 100% agree, however the system is actually highly structured. Despite a lack of mutual presence in an office, communication is highly prioritized.

  • Almost all planning is done via asynchronous chat rooms
  • Developers notify other developers directly using twitter-like @mentions
  • Software features are developed in pull-requests, which are a feature on github and other web-based distributed revision control systems.

There is constant communication, albeit non-synchronous. Developers can work with fewer distractions, checking into the chat rooms on their own schedule. Pull-requests (PR) are message-boards about individual git branches. PRs are cheap, developers are encouraged to open them often, and seek collaboration early.

More

Sounds great right? I recommend watching this talk by Zach Holman about How Github Uses Github to Build Github. Github has a very loose structure, but for those who need more control it seems like Kanban is a natural compliment to asynchronous development.

From reactions I can find on the web, people like this method. It empowers developers, and does a lot for morale and communication. Software development is one of the few careers that can be done very well in a distributed forum. I say leverage your ability to hire programmers anywhere in the world. Go distributed, go asynchronous!
